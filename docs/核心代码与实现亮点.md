## 核心代码与实现亮点

### 亮点一：基于数据驱动的 2.5D SVG 几何映射架构

在仪表盘可视化方案选型初期，我采用了 Three.js + WebGL 渲染管线加载高精度的 GLTF 人体肌肉模型。但在实际集成过程中暴露了两个突出问题：其一，模型文件达数 MB，首屏加载时间超过 3 秒，用户体验严重劣化；其二，引入 WebGL 运行时、控制器库及 GLTF 加载器，使前端总依赖体积膨胀至约 500KB，对于一个以数据看板为核心的系统而言，这是一种典型的过度设计。基于以上评估，我决定完全放弃 WebGL 渲染管线，转而设计了一套轻量级的 2.5D SVG 映射引擎。

该架构的核心思路是：将后端训练部位聚合数据，通过声明式的 `data-mesh` 自定义属性与 SVG 几何图形建立映射关系，再借助 CSS 3D 透视变换营造立体视觉。具体实现如下所示：

```javascript
// 数据驱动的高亮引擎：接受后端部位名称，驱动 SVG 状态切换
function highlightSvgMuscle(meshName) {
    document.querySelectorAll('.muscle-part').forEach(el => {
        el.classList.remove('active');
    });
    const targets = document.querySelectorAll(
        `.muscle-part[data-mesh="${meshName}"]`
    );
    targets.forEach(target => target.classList.add('active'));
}
```

```java
// 后端映射服务：根据训练动作查询应高亮的部位名称
public List<String> getHighlightMuscles(List<String> actions) {
    // 例：["卧推"] → ["Chest"]，映射由 ironsync.mesh.mappings 配置
    return actions.stream()
            .map(action -> meshMapping.get(action))
            .filter(Objects::nonNull)
            .collect(Collectors.toList());
}
```

前端 JS 只需五行核心逻辑即可完成后端数据到 SVG 部位的映射：`highlightSvgMuscle` 利用 CSS 属性选择器精确定位所有匹配 `data-mesh` 的几何元素，移除旧的 `active` 类后为当前目标添加发光动画。整个渲染管线无需 DOM 创建、没有虚拟 DOM 协调、不涉及 Canvas 绘制，完全利用浏览器原生 CSS 合成器完成过渡，从数据到达到视觉反馈的延迟控制在 16ms 以内。

在立体效果层面，容器层设置 `perspective: 1000px` 建立 3D 视场，SVG 根元素施加 `rotateX(10deg) rotateY(-15deg)` 的固定倾角，模拟全息投影面板的悬浮姿态。每一个 `.muscle-part` 在 hover 态下执行 `translateZ(5px) scale(1.06)`，配合 SVG `<linearGradient>` 定义的自左上至右下的金属渐变（`metal-grad`），在纯 2D 坐标系中营造出肌肉块面凹凸起伏的 2.5D 视觉效果。这实质上是一种用 CSS 变换模拟深度、用 SVG 渐变模拟材质的轻量化立体渲染技巧，在保持 60fps 流畅度的同时，实现了零第三方运行时依赖的秒级看板加载。

### 亮点二：基于状态模式的补剂生命周期管理

补剂库存预警模块的业务需求是：根据 `currentStockG / dailyConsumptionG` 计算的剩余天数，自动将补剂标记为"充足"（>30 天）、"偏低"（7-30 天）或"告急"（<7 天），并按严重程度排序。初版实现采用了多层 `if-else` 嵌套，预警阈值散落在业务方法中，维护成本高且极易出错。为此，我引入了有限状态机模式对库存状态判定进行重构。

核心设计包含两个类：`StatusLevel` 枚举和 `StatusMachine` 组件。`StatusLevel` 将三个状态编码为一组带谓词条件的枚举常量，每个状态持有自己的判定函数：

```java
public enum StatusLevel {
    SUFFICIENT("充足", d -> d.compareTo(new BigDecimal("30")) > 0),
    WARNING  ("偏低",  d -> d.compareTo(new BigDecimal("7")) >= 0),
    CRITICAL("告急",  d -> true);  // 兜底条件

    private final String displayName;
    private final Predicate<BigDecimal> condition;

    public static StatusLevel fromDays(BigDecimal days) {
        for (StatusLevel level : values()) {
            if (level.condition.test(days)) return level;
        }
        return CRITICAL;
    }
}
```

`StatusMachine` 作为 Spring 组件被 `SupplementServiceImpl` 组合使用，承担纯计算职责：

```java
@Component
public class StatusMachine {
    public String calculate(BigDecimal stockG, BigDecimal dailyConsumptionG) {
        BigDecimal days = stockG.divide(dailyConsumptionG, 2, RoundingMode.HALF_UP);
        return StatusLevel.fromDays(days).getDisplayName();
    }
}
```

这一设计的价值体现在三个层面。其一，**开闭原则**：新增状态（如"即将耗尽"）只需在 `StatusLevel` 中追加枚举常量及其阈值谓词，`StatusMachine` 无需修改。其二，**谓词即策略**：将条件判断封装为 `Predicate<BigDecimal>`，使阈值逻辑与枚举实例一一对应，消除了散落在业务代码中的魔法数字。其三，**编译器保障的安全性**：状态迁移路径由枚举迭代顺序隐式表达——`fromDays` 按 SUFFICIENT → WARNING → CRITICAL 的优先级依次测试谓词，保证任意输入都落入且仅落入一个状态，杜绝了 `if-else` 常见的手动遗漏兜底分支的问题。在此架构下，补剂模块的库存预警逻辑实现了完全的规则与数据分离，任何阈值的调整都不再触及业务事务方法，显著提升了模块的可维护性。
