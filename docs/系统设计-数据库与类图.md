## 数据库设计与类图

### 数据库 E-R 图

```mermaid
erDiagram
    user_profile ||--o{ training_record : "训练记录"
    user_profile ||--o{ supplement : "补剂库存"
    user_profile ||--o{ diet_mood : "饮食情绪"
    user_profile ||--o{ body_metrics : "身体指标"

    user_profile {
        bigint id PK "用户主键"
        decimal height_cm "身高(cm)"
        varchar goal "训练目标"
        datetime created_at "注册时间"
    }

    training_record {
        bigint id PK "记录主键"
        bigint user_id FK "所属用户"
        varchar action_name "动作名称"
        decimal weight_kg "使用重量(kg)"
        int reps "每组次数"
        int sets "组数"
        decimal rpe "RPE 自感用力度"
        date record_date "训练日期"
        tinyint deleted "逻辑删除标志"
    }

    supplement {
        bigint id PK "补剂主键"
        bigint user_id FK "所属用户"
        varchar name "补剂名称"
        decimal current_stock_g "当前库存总量(g)"
        decimal daily_consumption_g "每日消耗量(g)"
        varchar status "预警状态"
        datetime created_at "创建时间"
    }

    diet_mood {
        bigint id PK "记录主键"
        bigint user_id FK "所属用户"
        date record_date "记录日期"
        decimal carbs_g "碳水化合物(g)"
        decimal protein_g "蛋白质(g)"
        decimal fat_g "脂肪(g)"
        int mood_score "情绪评分(1-10)"
        varchar note "备注"
    }

    body_metrics {
        bigint id PK "记录主键"
        bigint user_id FK "所属用户"
        date record_date "记录日期"
        decimal weight_kg "体重(kg)"
        decimal body_fat_percentage "体脂率(%)"
        varchar note "备注"
    }
```

> **图注：** IronSync 系统核心 E-R 图。`user_profile` 作为根实体，通过 `user_id` 外键与四张业务表形成一对多关系。`training_record` 采用逻辑删除（`deleted` 标志位）；`diet_mood` 与 `body_metrics` 在 `(user_id, record_date)` 上建有唯一索引，保证每日一条的业务约束。

### 补剂状态机核心架构类图

```mermaid
classDiagram
    class SupplementController {
        - SupplementService supplementService
        + create(SupplementCreateDTO) Result~SupplementVO~
        + update(Long, SupplementUpdateDTO) Result~SupplementVO~
        + deleteById(Long) Result~Void~
        + findById(Long) Result~SupplementVO~
        + findAll() Result~List~SupplementVO~~
        + getStatus() Result~List~SupplementStatusVO~~
    }

    class SupplementService {
        <<interface>>
        + create(SupplementCreateDTO) SupplementVO
        + update(SupplementUpdateDTO) SupplementVO
        + deleteById(Long) void
        + findById(Long) SupplementVO
        + findAll() List~SupplementVO~
        + getStatusList() List~SupplementStatusVO~
    }

    class SupplementServiceImpl {
        - SupplementMapper supplementMapper
        - StatusMachine statusMachine
        + create(SupplementCreateDTO) SupplementVO
        + update(SupplementUpdateDTO) SupplementVO
        + deleteById(Long) void
        + findById(Long) SupplementVO
        + findAll() List~SupplementVO~
        + getStatusList() List~SupplementStatusVO~
        - toVO(Supplement) SupplementVO
        - toStatusVO(Supplement) SupplementStatusVO
        - statusOrder(String) int
    }

    class StatusMachine {
        + calculate(BigDecimal, BigDecimal) String
        + calcRemainingDays(BigDecimal, BigDecimal) BigDecimal
    }

    class StatusLevel {
        <<enumeration>>
        SUFFICIENT
        WARNING
        CRITICAL
        + fromDays(BigDecimal) StatusLevel
        + getDisplayName() String
    }

    class SupplementMapper {
        <<interface>> Mapper
        + insert(Supplement) int
        + selectById(Long) Supplement
        + update(Supplement) int
        + deleteById(Long) int
        + selectAll() List~Supplement~
    }

    SupplementController --> SupplementService : 依赖
    SupplementServiceImpl ..|> SupplementService : 实现
    SupplementServiceImpl --> SupplementMapper : 依赖
    SupplementServiceImpl --> StatusMachine : 依赖
    StatusMachine --> StatusLevel : 使用
```

> **图注：** 补剂状态机核心调用链。`SupplementController` 接收 HTTP 请求并委托至 `SupplementService` 接口；`SupplementServiceImpl` 实现业务逻辑，组合 `SupplementMapper` 完成数据持久化，并调用 `StatusMachine` 计算库存预警等级。`StatusMachine` 依赖 `StatusLevel` 枚举通过天数阈值（>30 天充足、7-30 天偏低、<7 天告急）判定状态，实现库存预警规则与业务逻辑的解耦。
