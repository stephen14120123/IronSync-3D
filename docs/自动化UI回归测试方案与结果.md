## 自动化 UI 回归测试方案与结果

### 回归测试策略选型

在项目初期，看板功能的回归验证依赖手工黑盒测试——每次迭代合并前，由开发人员逐一点击页面、录入表单、核对数据。这种方式的弊端在 Sprint 3 多次迭代后逐渐显现：人工测试覆盖面有限、重复执行成本高，且极易因遗漏边界条件导致缺陷流入主分支。为此，我引入了 Playwright 框架构建基于 CI/CD 流程的自动化 UI 持续回归测试体系。

Playwright 的核心优势在于：其一，基于 CDP（Chrome DevTools Protocol）直接控制浏览器引擎，无需 WebDriver 中间层，启动速度和指令延迟均优于 Selenium；其二，内置自动等待机制，可基于元素可见性、网络请求完成度等条件智能同步，大幅减少测试脚本中的显式 `sleep` 调用；其三，支持在无头模式下高并发执行测试用例，结合 `webServer` 配置在测试启动前自动编译并运行 Spring Boot 应用，实现了从服务端启动到浏览器验证的全链路自动化。这套体系的建立使得每次代码变更后的回归验证时间从人工的 20 分钟压缩至流水线的 3 分钟以内，且覆盖范围由关键路径扩展至全模块。

### 回归测试覆盖范围

测试套件按业务模块划分为三个 `test.describe` 块，覆盖系统核心功能的 CRUD 闭环、状态机跃迁及可视化渲染：

- **训练记录模块**（`training.spec.ts`）：验证表单创建 → 历史列表回显 → 删除操作的完整生命周期，同时检验 HTML5 表单校验在空提交场景下的拦截行为。
- **补剂管理模块**（`supplements.spec.ts`）：验证种子数据卡片渲染、新增补剂弹窗提交与列表刷新、编辑存量后的 Toast 反馈、搜索框实时过滤以及弹窗取消关闭等交互路径。
- **仪表盘模块**（`dashboard.spec.ts`）：验证 SVG 身体部位渲染、看板统计数据的 API 动态加载与回显、热量消耗环进度条渲染，以及导航栏页面跳转路由。

### 执行流程与预期结果

回归测试的执行流水线分为三个阶段。**前置数据准备**：`global-setup.ts` 在套件执行前通过 API 写入种子数据（包含今日训练记录、预置补剂），确保每个测试用例拥有确定的初始状态。**并发执行**：Playwright 以无头 Chromium 模式依次执行三个模块的测试用例。每个用例前通过 `gotoWithAuth` 经由 `/api/auth/login` 接口获取 JWT Token 并注入 `localStorage`，绕过前端登录页面的鉴权守卫。**断言校验**：通过 `toContainText`、`toBeVisible`、`not.toBeEmpty` 等语义化匹配器对页面状态进行声明式断言，CRUD 操作的成功与否通过 `waitForSuccessToast` 辅助函数捕获 Toast 组件的出现与消失来判定。
