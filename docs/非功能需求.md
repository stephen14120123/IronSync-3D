### 2.6 非功能需求

#### 2.6.1 性能需求

系统需在标准负载下保持稳定的响应能力。所有 RESTful API 接口的 95 百分位响应时间应控制在 500ms 以内，其中读密集型接口（如训练记录分页查询、看板摘要聚合）需在 300ms 内完成响应。服务端基于 MyBatis 参数化查询（`#{}` 预编译绑定）与 HikariCP 连接池的高效协作，通过预编译语句缓存减少 SQL 解析开销，连接池最大连接数预设 50，空闲回收时间严格控制在 300s 以内，避免数据库连接泄漏。前端 SVG 渲染层采用 `requestAnimationFrame` 驱动交互帧，所有 CSS 动画利用 GPU 合成层独立渲染，确保部位高亮切换与悬停浮出效果不阻塞主线程。系统采用**零第三方运行时依赖的原生 HTML/JS 架构**，无任何前端框架或库的加载开销，极大降低了首屏解析与执行耗时，页面完全加载时间控制在 1.5s 以内。面对并发峰值场景（如早间训练高峰），系统基于 Docker Compose 水平扩展能力支持至少 500 并发用户同时访问，超出连接池阈值时通过 HikariCP 的连接超时机制自动拒绝过量请求，配合 HTTP 429 状态码保护后端稳定性。

#### 2.6.2 安全需求

系统采用基于 JWT Token 的请求级鉴权机制，通过**自定义 Spring MVC Interceptor（`AuthInterceptor`）实现轻量级无状态拦截鉴权**，在 `preHandle` 阶段完成 Token 签名校验、过期判定与用户身份解析，所有非白名单接口（如登录、注册）均需携带有效 Token 方可访问。密码存储使用 BCrypt 算法加盐哈希，杜绝明文存储。数据访问层全部采用 MyBatis 参数化查询（`#{}` 预编译绑定），从根本上防御 SQL 注入攻击。前端输出侧对所有动态渲染内容执行 HTML 实体转义，防止反射型与存储型 XSS 攻击。跨域资源共享（CORS）策略限定为可信来源白名单，并关闭 HTTP 方法中的非必要谓词。敏感操作（如补剂消耗记录变更）需二次校验 Token 携带的用户身份与操作权限。此外，系统引入安全响应头（`X-Content-Type-Options: nosniff`、`X-Frame-Options: DENY`、`Content-Security-Policy` 等）加固浏览器端安全防线。日志输出侧通过过滤器统一替换 Token 与密码字段，避免凭据以明文形式写入日志文件。

#### 2.6.3 可用性需求

系统整体可用性目标为 99.9%（月度统计），对应月度不可用时间不超过 43 分钟。后端应用与 MySQL 数据库分离部署于 Docker 容器，通过 `depends_on` 与健康检查机制确保服务启动顺序正确，数据库故障时 HikariCP 连接池具备自动重连能力。前端采用响应式布局设计，基于 CSS Flexbox 和相对单位适配从平板（768px）到桌面宽屏（1920px+）的各类分辨率，在移动端浏览器中保持核心数据看板功能可用。系统内置**统一的全局异常处理器（`GlobalExceptionHandler`）**，所有 Controller 层抛出的业务异常与非预期运行时异常均被统一拦截，结合**标准的 `Result<T>` 响应体**封装为结构化错误信息返回前端——业务异常返回语义明确的状态码与错误消息，非预期异常以降级模式静默处理，确保高并发场景下的系统自我保护与优雅降级，避免堆栈信息泄露或主体功能瘫痪。非关键数据加载失败（如每周建议推荐接口超时）独立捕获处理，不影响训练记录、补剂管家等主功能呈现。

#### 2.6.4 扩展性与维护性需求

系统采用前后端分离架构，后端仅提供 RESTful API，前端通过 `api.js` 统一封装请求层，视图渲染与业务逻辑完全解耦。后端遵循分层架构设计（Controller — Service — DAO），业务逻辑收敛于 Service 层，数据库操作抽象于 MyBatis Mapper 接口，单个模块的替换或升级无需牵连上下游。SVG 身体部位映射采用声明式 `data-mesh` 属性驱动，新增或调整肌肉分区只需修改 HTML 模板与 CSS 样式，无需改动交互脚本，降低了前端可视化层的维护成本。配置文件与环境信息完全分离（`application.yml` / `application-dev.yml`），数据库连接、文件路径等基础设施参数支持通过环境变量覆盖，适配开发、测试、生产多环境部署。项目依赖声明集中管理于 `pom.xml`，前端无第三方运行时依赖，消除了 JavaScript 库版本冲突与升级阻塞的风险，为后续功能迭代保留了充分的技术灵活性。
